# 文件处理细则

## 编码与换行

- 默认按 UTF-8 读取和创建文本文件。
- 修改已有文件前，尽量识别当前编码、换行符和 BOM 状态。
- 修改已有文件时保持原编码、换行风格和文件权限，除非明确要求转换。
- 不确定编码时，先只读检查，不直接覆盖写入。
- 不为了统一风格而批量改变无关空白、排序、换行或文件权限。

## 读取与搜索

- 修改文件前先读取当前内容，必要时读取调用方、测试、配置和相邻文件。
- 文件名、路径或内容包含中文、空格或特殊字符时，使用能显式处理编码和字面路径的工具。
- 在 macOS / Linux 环境中，优先使用 `rg` 搜索；如果没有 `rg`，再使用系统自带工具。
- 简单查看文本可以使用 `cat`、`less`、`sed -n` 或编辑器预览；处理中文、编码转换或批量改写时，不依赖隐式编码。

## 修改原则

- 优先做补丁式小改动，不无故整文件重写。
- 不覆盖、回滚或删除用户已有改动，除非用户明确要求。
- 不把格式化、依赖升级、目录迁移和风格迁移混入当前任务，除非它们是完成任务所必需。
- 对生成文件、缓存文件、锁文件和依赖清单保持谨慎，只在任务需要且理解影响时修改。
- 大范围替换、批量迁移或格式转换前，先说明范围、风险和验证方式。

## 中文文本

- 中文文档和注释默认使用 UTF-8。
- 修改中文内容时保持原文语气和术语，除非任务目标就是改写风格。
- 不使用容易破坏编码或换行的临时脚本处理中文文件。
- 批量修改、格式转换或结构化文本处理时，使用 Python、Node.js、Go 或项目已有工具，并显式声明编码。

## 注释和文档

- 新增代码注释默认使用中文，除非项目已有风格、公开 API、协议字段、错误码或 CLI 帮助使用英文。
- 只在解释复杂逻辑、边界条件或非显然取舍时添加注释。
- 不为显而易见的赋值、返回或简单分支添加空泛注释。
- 面向团队共享的文档应避免个人化表达，优先记录事实、命令、约束和验证方式。
