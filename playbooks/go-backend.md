# Go Backend Playbooks

这些流程用于手动触发，不是每次对话都必须执行。先以团队仓库命令和公司规范为准，再参考本文件。

## 1. 读仓流程

目标：在不改代码的前提下建立仓库地图。

步骤：

1. 读取 `README`、构建脚本、CI 配置、`go.mod`、入口目录和测试目录。
2. 找到服务入口、配置加载、路由或 RPC 注册、核心业务模块、存储访问层。
3. 总结服务负责什么、不负责什么、主要上下游依赖是什么。
4. 标记仍需向同事确认的事实，不把推断写成确定结论。

输出格式：

```text
服务职责：
主要入口：
核心链路：
关键目录：
常用命令：
未确认问题：
```

## 2. 做需求流程

目标：小范围、可验证地完成一个后端改动。

最小实现阶梯：

1. 这个需求是否真的需要做，还是已有能力已经覆盖。
2. 仓库里是否已有 helper、类型、模式或相邻实现可以复用。
3. Go 标准库、数据库约束、系统能力或框架原生能力是否已经覆盖。
4. 已安装依赖是否能解决，且不会引入额外维护成本。
5. 只有以上都不成立时，新增最小代码。

步骤：

1. 明确目标、非目标、约束和完成标准。
2. 读取相关代码、测试、配置、调用方和类似实现。
3. 按最小实现阶梯选择方案，避免未请求的抽象、依赖、配置和脚手架。
4. 给出 3 到 6 个要点的修改计划；复杂、跨模块或生产风险较高时先等待确认。
5. 实施最小改动，遵循项目现有命名、错误处理、日志和测试风格。
6. 补充或更新测试。
7. 运行与风险匹配的验证命令。
8. 总结实际修改、验证结果和剩余风险。

常见验证候选：

```bash
go test ./...
go test -race ./...
go vet ./...
golangci-lint run
govulncheck ./...
```

执行哪些命令以项目已有工具链和团队规范为准。

## 3. 调试流程

目标：先定位事实，再提出修复。

步骤：

1. 读取错误输出、日志、复现步骤和相关测试。
2. 定位到具体输入、文件、函数、依赖或边界条件。
3. 查找相关调用方和共享路径，优先修根因，不只修报告中出现的单一路径。
4. 建立最小假设，一次只验证一个关键假设。
5. 修复后运行最小相关验证；通过后再考虑更大范围验证。
6. 如果同一错误反复失败，停止连续尝试，重新检查假设和上下文。

调试时优先保留：

- 状态码、错误码、请求 ID、时间范围、脱敏后的上下文。
- 触发条件、预期行为、实际行为。
- 最小复现方式。

调试时避免：

- 原样复述密钥、Cookie、完整连接串、用户隐私数据。
- 在没有证据时归因到某个模块或同事。
- 顺手重构无关代码。

## 4. PR 自查流程

提交前按以下顺序检查：

1. 正确性：是否覆盖目标需求，是否引入行为变化。
2. 最小性：是否复用了已有实现、标准库和平台能力，是否避免未请求的抽象、依赖和配置。
3. 边界条件：空值、超时、取消、重试、幂等、并发、资源关闭。
4. 错误处理：是否返回或包装错误，是否保持项目既有错误风格。
5. 日志和指标：是否足够定位问题，是否避免敏感信息。
6. 测试：非平凡逻辑是否留下至少一个能失败的检查，例如相关单测、集成测试或最小手动验证。
7. 兼容性：配置、数据结构、接口字段、灰度和回滚是否受影响。
8. 性能：是否新增循环调用、锁竞争、无界 goroutine、重复 IO 或大对象复制。
9. 安全：鉴权、权限、输入校验、依赖、网络请求和日志脱敏是否正确。

PR 描述建议包含：

```text
背景：
修改：
验证：
风险：
回滚：
```

## 5. 需要停下来问人的信号

- 生产数据、权限、鉴权、计费、风控、隐私或合规路径不明确。
- 发布、回滚、灰度、迁移、补偿方案不明确。
- 仓库中存在多个相互冲突的实现模式。
- 测试或 CI 失败原因无法通过当前上下文解释。
- 修改范围开始扩散到原目标之外。
- AI 给出的方案依赖未确认的内部 API、字段或流程。
